git commit规范

发表于 2018-07-16 | 分类于 技术总结
字数统计: 561 | 阅读时长 ≈ 2

多人协作的项目，在每次提交变动时，建议遵循Git Commit Message规范。好处多多，具体自查并且建议每次一个小的更改就提交一次。

Git Commit Message格式：

<type>(<scope>): <subject>

  • type（必填）

    1. feat：增加的新功能或者新需求
    2. fix：修复bug
    3. docs：文档的变动
    4. style： 代码格式以及页面样式等等不影响代码运行的变动
    5. refactor：重构（即不是新增功能，也不是修改bug的代码变动）
    6. test：增加测试代码或者修改测试代码
    7. chore：构建过程或辅助工具的变动

  • scope（选填）

    用于说明 commit 影响的范围，比如数据层、控制层、视图层等等，视项目不同而不同。(官方说法，但是目前看来在我们所涉及的项目中建议不写。)

  • subject（必填）

    本次的提交的大概描述。建议不超过50个字符。

    注意：一个空格后再写描述。

    墙裂建议！！！！在多人的协作项目中统一要求用中文！！！

  • 栗子：

    1. fix: 修复XXXXX导致的BUG（#7777）

      如果有bug单，需要在末尾加上bug单，用来追踪详细的bug。

    2. feat: 新增XXXXXX功能
