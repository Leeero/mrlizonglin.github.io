mpvue脱坑指北「一」

发表于 2019-11-19 | 分类于 框架
字数统计: 1k | 阅读时长 ≈ 3

mpvue是一个使用vue.js开发小程序的前端框架，非常适合会vue但是没有开发过小程序的猿。but，它跟其他开源框架一样存在各种各样的坑，所以总结一下，避免重复掉坑。顺便也吐槽一下这个开源团队对于某些问题的解决态度~

关于长列表性能优化

问题：

在实际的业务开发中经常会有长列表的业务场景，大家可能也遇到过当列表加载几次之后整个小程序会变得异常卡顿，严重的甚至会造成小程序卡死退出等问题，所以对于长列表的性能优化是一个不得不做的事情。

原因：

我们在实际的开发中写在data、props、computed里的数据，在mpvue编译成小程序代码后，每次数据的变动都是调用小程序的setData方法将数据序列化成字符串之后写入到 JSRender 进程中，众所周知JS是单线程的，所以当每次调用setData时需要处理的数据量过大，就会形成阻塞，那么卡顿肯定就一定会出现的。

其实数据的变动只能调用setData方法这一点是微信限制的，不管是mpvue还是taro，亦或者是uniapp都是这样处理的。但是mpvue很坑的一点是在于它每次是全量去更新数据的。什么是全量更新呢，其实就是你某次的数据更新了data里的某一个值，但是mpvue是直接更新了整个data里的所有数据，这无疑是非常不合理的。

调试：

在开发者工具上运行小程序，然后调试器栏目找到Sources，打开common下的vendor.js。然后找updateDataToMP方法，打上断点，就可以看到你每次操作时数据变动之后调用setData方法变更的数据量了。

解决：

在mpvue的版本升级记录里1.0.18有用户提交了一个pr修复了这个问题“在渲染时使用新的diff算法来减少Vue组件更新引起的Page.setData的实际更新量，达到提升页面性能的目的”。在看到这次提交后，立马将项目的mpvue版本改为1.0.18，在升级完版本后，可以肉眼可见的发现长列表的性能得到了优化。在断点调试的时候可以发现数据更新已经由之前的全量更新改为现在的“部分”增量更新。为什么说是部分呢，因为在我的项目里我有用到mixin，我发现每次的setData还是不可避免的更新了mixin里的数据。总结一下长列表性能优化的方案吧：

  1. 将mpvue的版本切换至1.0.18。为什么一定是1.0.18呢，因为在某次的版本升级之后我发现setData又变成了全量更新！！！也就是说mpvue团队在某次的代码合并或者版本升级中将1.0.18的修复弄丢了，或者其他的原因，感兴趣的可以追一下commit看看是哪个版本之后出现的问题。
  2. 尽量不用或者少用mixin。因为在修复的版本中，每次的setData还是会同步去更新mixin里的数据，所以如果你的mixin里有过多的数据的话，那么肯定还是有性能问题。
  3. 将data里数据颗粒化。其实就是尽量将数据拆分成更多的key，而不是使用对象的形式。因为这样可以再一定程度上较少每次更新的数据量。
  4. 减少写入data里的数据。从后端拿到数据之后，对数据进行处理优化，将视图层不需要的数据删除不再一次性写入到data。
  5. 切换列表不需要实时更新时，将已经加载的列表用css的display属性隐藏而不是使用v-if。避免数据的重复加载。

以上，是我的个人实践，如果有更好的解决方案或者看法，请不吝赐教。
